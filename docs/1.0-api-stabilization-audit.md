# 1.0 API stabilization audit

Date: 2026-05-12
Scope: public package surface for `@opsydyn/elysia-spectral` before `1.0`

## Goal

Establish which APIs should be treated as stable for `1.0`, which APIs are advanced but supportable, and which exports are currently accidental or under-documented.

## Entry points under audit

The published package currently exposes exactly two entry points:

- `@opsydyn/elysia-spectral`
- `@opsydyn/elysia-spectral/core`

That is a good starting boundary for `1.0`: small, explicit, and easy to document.

## Current root surface

Current root exports include:

- plugin/runtime:
  - `spectralPlugin`
  - `createOpenApiLintRuntime`
- ruleset and lint helpers:
  - `loadRuleset`
  - `loadResolvedRuleset`
  - `lintOpenApi`
- presets:
  - `recommended`
  - `server`
  - `strict`
  - `presets`
- threshold helpers:
  - `shouldFail`
  - `enforceThreshold`
- error classes:
  - `OpenApiLintThresholdError`
  - `OpenApiLintArtifactWriteError`
  - `RulesetLoadError`
- curated public types from `src/types.ts`
- selected ruleset-loader types re-exported from `./core`

## Current `./core` surface

Current `@opsydyn/elysia-spectral/core` exports include:

- `lintOpenApi`
- `loadRuleset`
- `loadResolvedRuleset`
- `defaultRulesetResolvers`
- ruleset loader types
- `RulesetLoadError`
- `createOpenApiLintRuntime`
- `OpenApiLintArtifactWriteError`
- `shouldFail`
- `enforceThreshold`
- `OpenApiLintThresholdError`

This subpath is already acting as the advanced/programmatic API.

## Observed consumer usage in this repo

Observed downstream usage is narrow:

- `apps/dev-app` uses `spectralPlugin` from the root entry point.
- README examples use:
  - `spectralPlugin`
  - `strict`
  - `createOpenApiLintRuntime`
  - `loadRuleset` from `@opsydyn/elysia-spectral/core`
- packed smoke tests now assert the approved root and core runtime export surfaces, including `presets`, threshold helpers, and `defaultRulesetResolvers`

No repo-local production consumer currently depends on undocumented `./core` details such as `defaultRulesetResolvers`, but external consumers still might.

## Findings

### 1. Entry-point boundaries are already in decent shape

Keeping the package to two published entry points is a strong base for `1.0`.

Recommendation:

- keep `.` as the primary supported consumer surface
- keep `./core` as the advanced/programmatic surface
- avoid adding more public subpaths unless a strong packaging need appears

### 2. Root wildcard type exports were the biggest stabilization risk

At the start of this audit, the root entry point re-exported every type from `src/types.ts` via a wildcard export.

Why that matters:

- any new exported type added to `src/types.ts` automatically becomes public API
- removing or renaming a type later becomes a semver concern even if it was never intentionally documented
- the root API is harder to reason about because the public type surface is implicit rather than curated

Action taken during this audit:

- the root entry point now uses an explicit named type export list instead of `export * from './types'`
- a compile-time public type smoke file was added so the documented root types must continue to resolve during `tsc --noEmit`

Remaining recommendation before `1.0`:

- decide which types are root-safe and which should be `./core`-only

### 3. `./core` now has an explicit advanced support story

`loadRuleset`, `loadResolvedRuleset`, and `lintOpenApi` are sensible advanced APIs.

`defaultRulesetResolvers` is different:

- it exposes internal composition structure rather than a purely user-goal-oriented API
- changing resolver ordering or composition later could become a breaking change if this export is blessed implicitly

Action taken during this audit:

- the README now documents `defaultRulesetResolvers` as an advanced-but-supported export from `@opsydyn/elysia-spectral/core`
- unit tests now cover the recommended `[customResolver, ...defaultRulesetResolvers]` composition pattern

Recommendation before `1.0`:

- preserve resolver ordering semantics unless a major version intentionally changes the composition contract

### 4. Root threshold helpers need an intentional support decision

The root entry point exports:

- `shouldFail`
- `enforceThreshold`
- `OpenApiLintThresholdError`

These are useful and tested, but they are closer to supporting utilities than to the core plugin story.

Action taken during this audit:

- `shouldFail` and `enforceThreshold` are now documented as part of the stable root utility surface
- packed-package smoke tests now assert that the root and `./core` entry points both expose the threshold helpers

Recommendation before `1.0`:

- keep their behavior aligned across root and `./core` exports

### 5. `presets` is exported but under-documented

The individual presets are documented and clearly part of the product.

The aggregate `presets` object is exported from the root, but the README focuses on the named preset exports. That makes `presets` feel slightly accidental.

Action taken during this audit:

- the README now documents `presets` as a stable root export alongside the named presets
- packed-package smoke tests assert that the root export resolves to an object in packed consumers

Recommendation before `1.0`:

- treat `presets` as the supported preset map keyed by `PresetName`

### 6. Plugin runtime access via `app.store.openApiLint` is documented, but its support level should be made explicit

The README documents that the runtime is available on `app.store.openApiLint` when used as a plugin.

That is valuable, but `1.0` should explicitly decide whether this is:

- a first-class supported integration point, or
- a convenience that may evolve with Elysia typing/runtime behavior

Action taken during this audit:

- the README now states explicitly that `app.store.openApiLint` is part of the supported plugin contract

Recommendation before `1.0`:

- keep integration coverage around the runtime store attachment

### 7. Environment/runtime support policy is explicit, but should be frozen for `1.0`

The package already declares:

- Node `>=22.0.0`
- Bun `>=1.3.0`

Recommendation before `1.0`:

- keep these engine constraints explicit in the `1.0` support policy
- call out that the package is ESM-only
- keep packaged import smoke tests as release-gating coverage

### 8. `@elysiajs/openapi` dependency expectations are now documented clearly enough

The package intentionally depends on a served OpenAPI JSON route rather than a hard peer dependency on `@elysiajs/openapi`.

That flexibility seems worth preserving for `1.0`.

Recommendation:

- keep the current design
- keep the README language explicit that `@elysiajs/openapi` is the default/recommended generator, not the only supported source

## Backwards compatibility and default behavior notes

The current defaults and legacy mappings that should be treated as stable unless a major version intentionally changes them are:

- startup mode defaults to `enforce`
- `failOn` defaults to `error`
- artifact writes default to warning-only behavior
- `enabled: false` remains a backwards-compatible alias for `startup.mode: 'off'`
- the legacy `enabled(env)` callback maps to `enforce` / `off`
- `startup.mode` takes precedence over `enabled`

`startup.mode: 'report'` should be documented as a threshold-relaxation mode, not a blanket suppression mode for every startup error. Provider, ruleset, and fatal artifact-write failures remain actionable startup errors.
