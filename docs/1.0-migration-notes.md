# 1.0 migration notes

Date: 2026-05-12
Audience: teams upgrading from early `v0.1`-style or other pre-`1.0` `@opsydyn/elysia-spectral` usage to the stable `v1` contract.

## TL;DR

Most simple plugin setups continue to work.

If you only use:

- `spectralPlugin({...})`
- a public OpenAPI JSON route (usually `@elysiajs/openapi`)
- `failOn`
- `ruleset` or `preset`
- optional `healthcheck` / `output`

then migration is mostly about adopting the newer preferred configuration style and updating any automation that reads lint result artifacts.

The main upgrade points are:

1. prefer `startup.mode` over the legacy `enabled` flag
2. treat `LintRunResult.ok` as threshold-aware, not just `error === 0`
3. resolve `result.artifacts.*` paths relative to `process.cwd()`
4. use only the documented root and `./core` import surfaces
5. opt into `healthcheck` and `dashboard` explicitly when you want routes exposed

## Quick migration checklist

- [ ] Replace new code that uses `enabled` with `startup.mode`
- [ ] If you read `result.ok`, confirm that threshold-aware semantics are what you want
- [ ] If you open artifact files from `result.artifacts`, resolve relative paths with `path.resolve()`
- [ ] Remove imports of undocumented helpers such as `normalizeFindings`, `isEnabled`, `exceedsThreshold`, or `SpecProvider`
- [ ] Explicitly configure `healthcheck` / `dashboard` routes instead of assuming route exposure
- [ ] In CI, set `output.artifactWriteFailures: 'error'` if artifacts are required outputs

## Preferred configuration style in `v1`

### Startup control

Legacy configuration still supported:

```ts
spectralPlugin({
  enabled: false,
})
```

Preferred `v1` style:

```ts
spectralPlugin({
  startup: {
    mode: 'off',
  },
})
```

For non-blocking startup lint output in local development:

```ts
spectralPlugin({
  failOn: 'warn',
  startup: {
    mode: 'report',
  },
})
```

Important behavior note:

- `startup.mode: 'report'` only relaxes **threshold failures**
- missing OpenAPI routes, invalid spec JSON, broken rulesets, or fatal artifact-write errors still fail startup

Compatibility contract:

- default startup mode is `enforce`
- `enabled: false` remains a backwards-compatible alias for `startup.mode: 'off'`
- `enabled(env)` still maps to `enforce` / `off`
- `startup.mode` takes precedence when both are provided

### Ruleset strategy

Older code may pass a raw ruleset path or object directly:

```ts
spectralPlugin({
  ruleset: './spectral.yaml',
})
```

That still works.

Preferred `v1` style for most services:

```ts
spectralPlugin({
  preset: 'strict',
})
```

Then add repo-local policy overrides in `spectral.yaml` / `spectral.ts` as needed.

That gives you:

- a stable first-party baseline (`recommended`, `server`, or `strict`)
- local service overrides without forking the package defaults
- clearer upgrade semantics as presets evolve intentionally

For advanced programmatic ruleset composition, use `@opsydyn/elysia-spectral/core` and `defaultRulesetResolvers`.

## Public import surface changes

Treat these as the stable import surfaces going forward.

### Root entry point: `@opsydyn/elysia-spectral`

Use the root entry point for normal application and CI usage:

- `spectralPlugin`
- `createOpenApiLintRuntime`
- `recommended`, `server`, `strict`, `presets`
- `loadRuleset`, `loadResolvedRuleset`, `lintOpenApi`
- `shouldFail`, `enforceThreshold`
- public types and error classes documented in the README

### Advanced entry point: `@opsydyn/elysia-spectral/core`

Use `./core` for advanced composition details:

- `defaultRulesetResolvers`
- resolver pipeline types
- programmatic lint/runtime helpers when you want the explicitly advanced surface

### No longer part of the supported public contract

Do not depend on these as public API anymore:

- `normalizeFindings`
- `isEnabled`
- `exceedsThreshold`
- `SpecProvider`

If older code imported them directly, migrate to the supported APIs instead:

- use `lintOpenApi(...)` instead of `normalizeFindings(...)`
- use `shouldFail(...)` / `enforceThreshold(...)` instead of internal threshold helpers
- use `startup.mode` / `enabled` compatibility instead of importing startup decision helpers other than the runtime-owned behavior
- treat providers as internal runtime details; configure `source.specPath` / `source.baseUrl` instead

## Result model changes

The result model is richer than the early `v0.1` shape.

`LintRunResult` now includes stable metadata such as:

- `source`
- `failOn`
- `durationMs`
- `artifacts`

### `ok` is now threshold-aware

If your old code treated `result.ok` as equivalent to `result.summary.error === 0`, review that assumption.

Current behavior:

- `result.ok` is `true` when findings stay below the configured `failOn` threshold
- with `failOn: 'warn'`, warnings can make `ok === false`
- with `failOn: 'never'`, `ok` can remain `true` even when findings exist

If your automation only cares about error counts, inspect `result.summary.error` directly.

### Artifact paths are relative in `v1`

This is the main declared `1.0` breaking change.

Current behavior:

- `result.artifacts.*` paths are relative to `process.cwd()`
- persisted JSON reports also store relative artifact paths

If older code used the artifact path directly as if it were absolute, update it like this:

```ts
import path from 'node:path'

const artifactPath = result.artifacts?.jsonReportPath
const absolutePath = artifactPath
  ? path.resolve(process.cwd(), artifactPath)
  : null
```

## Route exposure and runtime access

### Healthcheck and dashboard are explicit opt-ins

Do not assume route exposure from enabling linting alone.

Configure them intentionally:

```ts
spectralPlugin({
  healthcheck: { path: '/health/openapi-lint' },
  dashboard: { path: '/__openapi/dashboard' },
})
```

### Plugin runtime access is supported

When the plugin is mounted, the runtime is available on:

```ts
app.store.openApiLint
```

That is now part of the supported plugin contract.

## CI migration guidance

If you previously drove linting indirectly through app startup, the preferred `v1` CI workflow is the exported runtime:

```ts
import { createOpenApiLintRuntime } from '@opsydyn/elysia-spectral'
import { app } from '../src/app'

const runtime = createOpenApiLintRuntime({
  preset: 'strict',
  failOn: 'error',
  output: {
    console: true,
    jsonReportPath: './reports/openapi-lint.json',
    sarifReportPath: './reports/openapi-lint.sarif',
    junitReportPath: './reports/openapi-lint.junit.xml',
    specSnapshotPath: './reports/openapi-snapshot.json',
    artifactWriteFailures: 'error',
  },
})

await runtime.run(app)
```

Recommended CI stance:

- keep `failOn: 'error'` or stricter as needed
- set `artifactWriteFailures: 'error'` when reports/snapshots are required deliverables
- upload SARIF/JUnit artifacts even on failure

## Recommended target config for most services

If you want a stable landing zone after migration, this is a good default target:

```ts
import { openapi } from '@elysiajs/openapi'
import { spectralPlugin } from '@opsydyn/elysia-spectral'
import { Elysia } from 'elysia'

new Elysia()
  .use(openapi())
  .use(
    spectralPlugin({
      preset: 'strict',
      failOn: 'error',
      startup: { mode: 'enforce' },
      healthcheck: { path: '/health/openapi-lint' },
      output: {
        console: true,
        jsonReportPath: './artifacts/openapi-lint.json',
        specSnapshotPath: true,
      },
    }),
  )
```

For local development or intentionally broken fixtures, switch only the startup mode:

```ts
startup: { mode: 'report' }
```

## If you are upgrading from a very early internal `v0.1` snapshot

The biggest conceptual changes since the earliest prototype are:

- lint route exposure is no longer implicit; healthcheck/dashboard are opt-in
- runtime execution is a first-class exported API (`createOpenApiLintRuntime`)
- presets are now the normal top-level governance story
- output/reporting is sink-based under the hood, with stable convenience options on top
- result artifacts are self-describing and portable across machines
- only documented root and `./core` exports should be treated as public API
