this plan assumes four facts: Elysia’s OpenAPI plugin exposes a JSON spec endpoint via specPath (defaulting under the docs path), Elysia plugins are composed with use(), Spectral has a JavaScript API plus built-in OAS rulesets, and Bruno can import OpenAPI v3 into a generated collection artifact.  ￼

# elysia spectral plugin — design planning spec

## status

proposed

## owner

platform / api tooling

## summary

build a small Elysia plugin that lints the generated OpenAPI document with Spectral.

the plugin must treat Elysia route schemas as the source of truth, run Spectral against the generated OpenAPI document, and expose the results in a way that is useful for local development, CI, and downstream tooling.

the plugin must stay thin:
- no duplicate API contract model
- no custom spec builder
- no attempt to replace `@elysiajs/openapi`
- no attempt to become a full API governance platform

## problem

today we can generate OpenAPI from Elysia, but we do not have a first-class way to enforce contract quality from inside the app boundary.

we want:
- one source of truth
- fast local feedback
- CI-friendly contract checks
- optional machine-readable reports
- a clean path to downstream generation such as Bruno / Postman / client SDKs

without this, teams drift into:
- undocumented style issues
- inconsistent operation metadata
- weak error model descriptions
- missing examples / tags / summaries
- ad hoc API review during PRs

## goals

### primary goals

1. lint the OpenAPI document generated by Elysia using Spectral
2. support startup-time linting
3. support optional on-demand lint reporting for local development
4. support configurable failure thresholds
5. support custom rulesets
6. support machine-readable report output
7. make the result usable in CI

### secondary goals

1. make it easy to chain into collection generation after lint passes
2. provide pleasant terminal output in local development
3. provide typed plugin options
4. keep the implementation Bun-first but portable where practical

## non-goals

1. generating OpenAPI ourselves
2. replacing `@elysiajs/openapi`
3. mutating the OpenAPI doc to “fix” issues automatically
4. building a general-purpose API design UI
5. shipping Bruno or Postman generation in v1
6. supporting every Spectral feature on day one
7. linting arbitrary remote specs in production

## user stories

### local developer

as a developer,
i want OpenAPI lint failures to show up during startup,
so that i catch contract problems immediately.

### CI pipeline

as a CI pipeline,
i want a deterministic JSON report and non-zero exit behaviour,
so that I can fail builds on contract issues.

### platform engineer

as a platform engineer,
i want to provide a shared ruleset,
so that all services follow the same API design standard.

### api consumer

as an API consumer,
i want route metadata to be consistent,
so that generated docs, collections, and clients are trustworthy.

## proposed solution

create a plugin package, tentatively:

`@your-scope/elysia-spectral`

usage:

```ts
import { Elysia } from 'elysia'
import { openapi } from '@elysiajs/openapi'
import { spectralPlugin } from '@your-scope/elysia-spectral'

new Elysia()
  .use(openapi())
  .use(
    spectralPlugin({
      ruleset: './spectral.yaml',
      failOn: 'error',
      output: {
        console: true,
        jsonReportPath: './artifacts/openapi-lint.json'
      }
    })
  )

architecture

core idea

the plugin runs after the OpenAPI plugin is registered.

it obtains the generated OpenAPI document from the in-process app state if possible.

if that is not possible in a stable way, it can fall back to fetching the local spec endpoint, but that is a fallback only and must not be the primary design.

flow

elysia routes + schemas
    ->
@elysiajs/openapi generates openapi doc
    ->
elysia-spectral resolves doc
    ->
spectral lints doc against ruleset
    ->
results are formatted / emitted
    ->
optional failure on threshold
    ->
optional report written to disk

design principles
	1.	source of truth stays in Elysia schemas
	2.	linting is read-only
	3.	local dev ergonomics matter
	4.	CI output must be deterministic
	5.	production runtime cost should be near-zero unless explicitly enabled
	6.	plugin must degrade clearly when misconfigured
	7.	default behaviour should be safe and unsurprising

plugin API

v0.1 contract

this spec defines v0.1 only.

included in v0.1:
	•	startup lint
	•	threshold-based startup failure
	•	ruleset loading from local YAML path or in-memory object
	•	console summary output
	•	JSON report output
  •	optional OpenAPI snapshot output in the consuming app root

deferred beyond v0.1:
	•	manual lint route
	•	SARIF output
	•	JS/TS ruleset loading
	•	result filtering
	•	experimental in-process OpenAPI resolution

types

export type SeverityThreshold = 'error' | 'warn' | 'info' | 'hint' | 'never'

export type SpectralPluginOptions = {
  /**
   * local YAML ruleset path, imported ruleset object,
   * or undefined to use the package default ruleset
   */
  ruleset?: string | Record<string, unknown>

  /**
   * threshold that causes startup failure
   */
  failOn?: SeverityThreshold

  /**
   * output controls for v0.1
   */
  output?: {
    console?: boolean
    jsonReportPath?: string
    specSnapshotPath?: string | true
    pretty?: boolean
  }

  /**
   * public-surface document resolution settings
   */
  source?: {
    /**
     * path of the OpenAPI JSON endpoint exposed by @elysiajs/openapi
     */
    specPath?: string

    /**
     * base URL used when loopback HTTP fetch is required
     */
    baseUrl?: string
  }

  /**
   * environment guard
   */
  enabled?: boolean | ((env: Record<string, string | undefined>) => boolean)

  /**
   * custom logger adapter
   */
  logger?: {
    info: (message: string) => void
    warn: (message: string) => void
    error: (message: string) => void
  }
}

functional requirements

FR-1: startup lint

when enabled, the plugin must lint the generated OpenAPI document once during application startup.

FR-2: failure threshold

when findings meet or exceed failOn, the plugin must fail startup with a useful error summary.

FR-3: spec resolution contract

v0.1 must consume the public OpenAPI JSON document exposed by @elysiajs/openapi.

v0.1 must not depend on private OpenAPI plugin internals.

if a stable same-process request path exists, the adapter may use it.

otherwise, the plugin must use loopback HTTP with an explicit baseUrl.

FR-4: output artifacts

when jsonReportPath is set, the plugin must write findings to disk as JSON.

when specSnapshotPath is set, the plugin must write the generated OpenAPI document itself to disk as JSON.

when specSnapshotPath is `true`, the plugin must derive `./<package-name>.open-api.json` from the consuming app's package.json name, sanitized for filesystem use.

SARIF is deferred beyond v0.1.

FR-5: ruleset loading

the plugin must support:
	•	local YAML ruleset path
	•	direct ruleset object

FR-6: default ruleset

if no ruleset is provided, the plugin should use a minimal default based on Spectral OAS rules, then layer a few opinionated checks that are low-noise.

FR-7: clear diagnostics

output must include:
	•	finding count by severity
	•	rule code
	•	JSONPath / pointer
	•	human-readable message
	•	path + method when derivable

FR-8: reusable core engine

the core lint logic must be exported separately from the Elysia runtime plugin so CI and tests can invoke the same behavior without booting a server.

non-functional requirements

NFR-1: startup overhead

for a medium service, startup lint should feel acceptable for local development.
target: sub-second to low-single-digit seconds where possible.

NFR-2: determinism

the same OpenAPI input and ruleset must produce the same result set.

NFR-3: low coupling

the plugin must depend on stable public behaviour where possible.
avoid deep reliance on private Elysia internals unless wrapped behind a narrow adapter.

NFR-4: testability

core lint logic must be testable without booting an HTTP server.

NFR-5: environment safety

runtime lint execution and report writing must be explicitly controllable for production environments.

doc resolution strategy

v0.1 decision

resolve the generated OpenAPI document through the public JSON surface exposed by @elysiajs/openapi.

this keeps the plugin coupled to documented behavior rather than private in-process state.

provider contract:

interface SpecProvider {
  getSpec(): Promise<unknown>
}

ship in v0.1:
  •	PublicSpecProvider

behavior:
  •	use the configured specPath, or the OpenAPI plugin default path
  •	use Elysia's public app.handle(Request) API to request the OpenAPI JSON endpoint in-process
  •	synthesize a fully qualified local URL such as http://localhost${specPath} for the Request object
  •	use loopback HTTP only as a fallback if an adapter-specific runtime prevents app.handle from reaching the generated route as expected
  •	fail clearly if the public spec endpoint cannot be resolved

deferred beyond v0.1:
  •	InProcessSpecProvider backed by private plugin internals
  •	AutoSpecProvider that switches strategies implicitly

non-goal for v0.1:
  •	probing Elysia or @elysiajs/openapi internals to discover undocumented state

result model

internal normalized shape:

type LintSeverity = 'error' | 'warn' | 'info' | 'hint'

type LintFinding = {
  code: string
  message: string
  severity: LintSeverity
  path: Array<string | number>
  documentPointer?: string
  source?: string
  range?: {
    start?: { line: number; character: number }
    end?: { line: number; character: number }
  }
  operation?: {
    method?: string
    path?: string
    operationId?: string
  }
}

type LintRunResult = {
  ok: boolean
  generatedAt: string
  summary: {
    error: number
    warn: number
    info: number
    hint: number
    total: number
  }
  findings: LintFinding[]
}

default rule posture

v1 should be conservative.
prefer high-signal rules that improve docs and client generation without causing revolt.

candidate defaults:
	•	operation must have summary
	•	operationId should exist
	•	tags should exist
	•	error responses should be documented
	•	path params must match template params
	•	examples encouraged but not mandatory
	•	no empty descriptions where present
	•	servers block optional depending on deployment style

do not enable noisy style rules by default unless they are clearly valuable.

lifecycle integration

startup mode

the plugin runs once during startup after routes and the OpenAPI endpoint are in place.

if lint findings meet the configured threshold, startup fails with a concise summary.

manual mode

deferred beyond v0.1.

core mode

the same lint engine must be invokable directly from tests and CI without booting the runtime plugin.

error handling

misconfiguration cases
  1.	public OpenAPI endpoint not available
	•	fail fast with a clear message
	2.	ruleset path not found
	•	fail fast during startup
	3.	spec could not be resolved
	•	fail with provider-specific detail
	4.	report path unwritable
	•	configurable:
	•	warn only by default
	•	optional strict mode later
	5.	spectral internal error
	•	fail clearly with underlying message preserved

observability

the plugin should emit a small amount of structured log data:
	•	lint start
	•	lint complete
	•	counts by severity
	•	report path written
	•	startup blocked due to threshold

optional later:
	•	span around lint execution if OpenTelemetry is present

security considerations
	1.	do not expose lint routes in production by default
	2.	do not fetch arbitrary remote specs
  3.	write reports only to explicit configured output paths, except for the explicit opt-in `specSnapshotPath: true` mode which may derive a host-root filename from the consuming app package name
	4.	avoid leaking secrets through logged examples or server URLs
	5.	ensure loopback HTTP fallback cannot be abused as SSRF

acceptance criteria
	1.	plugin works with an Elysia app already using @elysiajs/openapi
  2.	startup lint runs successfully against the generated public OpenAPI document fetched through Elysia's public app.handle(Request) API

  4.	custom YAML ruleset file or imported ruleset object is supported

  4.	v0.1 ships JSON artifact output for lint findings and optional spec snapshots; SARIF is deferred
  5.	v0.1 keeps findings in memory only inside the runtime plugin; persistence is limited to explicit report or snapshot output

example pseudocode

export const spectralPlugin = (options: SpectralPluginOptions = {}) => {
  return new Elysia({ name: 'elysia-spectral' })
    .decorate('openApiLint', {
      latest: null as LintRunResult | null,
      async run(app: Elysia): Promise<LintRunResult> {
        const provider = createSpecProvider(app, options)
        const spec = await provider.getSpec()

        if (options.output?.specSnapshotPath) {
          await writeSpecSnapshot(resolveSnapshotPath(options.output.specSnapshotPath), spec)
        }

        const ruleset = await loadRuleset(options.ruleset)
        const findings = await lintOpenApi(spec, ruleset)

        const result = normalizeFindings(findings)

        this.latest = result

        if (options.output?.jsonReportPath) {
          await writeJsonReport(options.output.jsonReportPath, result)
        }

        if (options.output?.console !== false) {
          reportToConsole(result, options)
        }

        enforceThreshold(result, options.failOn ?? 'error')
        return result
      }
    })
    .onStart(async ({ app }) => {
      if (isEnabled(options)) {
        await app.openApiLint.run(app)
      }
    })
}
